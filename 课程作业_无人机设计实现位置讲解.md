# Milk-V Duo 无人机飞控技术实现复盘

这份说明按我当前实际项目来写，不把它说成已经完整稳定飞行。现在项目处在无桨台架验证阶段：遥控、IMU、磁力计、PWM、电机映射、安全门控、debug 输出都已经打通；姿态解算、互补滤波、完整 PID、悬停、低电压报警和功能传感器可视化还没有完全完成。

## 1. 当前总体进度

已经完成：

- Milk-V Duo 256M / SG2002 作为主控，C906 小核跑 FreeRTOS/CVIRTOS。
- GP5 / UART3_RX 读取 FS-iA6B iBUS。
- GP0/GP1 用 GPIO bit-bang I2C 读取 HMC5883L 和 MPU6050。
- 四路 PWM 已经能驱动电调和电机。
- 当前 motor map 已确认：

```text
motor0 = LF 左前 -> GP2 -> PWM_7 -> pwm_id=1 ch=3
motor1 = RF 右前 -> GP4 -> PWM_5 -> pwm_id=1 ch=1
motor2 = RR 右后 -> GP9 -> PWM_4 -> pwm_id=1 ch=0
motor3 = LB 左后 -> GP3 -> PWM_6 -> pwm_id=1 ch=2
```

注意：GP9 是 PWM_4，不是 PWM_9。README 早期的 PWM10/PWM11 不能直接套到现在的接线。

已经做出的控制链路：

```text
IBUS 摇杆 -> setpoint -> body_rate feedback -> rate P 控制
          -> angle assist -> mixer -> motor trim/min_spin -> PWM
```

还没完成：

- IMU 采样现在是 `20ms`，约 50Hz，还没达到作业要求的 100Hz。
- 当前姿态不是完整互补滤波。
- 当前 PID 是 P-only rate controller + angle assist，不是完整位置/速度级联 PID。
- 没有低电压检测。
- 没有额外功能传感器可视化。
- 没有完成有桨稳定悬停 10 秒验证。

## 3. IBUS 遥控输入

代码位置：

```text
comm_main.c
uart3_read_byte()          读取 UART3 单字节
ibus_parse_byte()          解析 32 字节 iBUS 帧
ibus_probe_poll_once()     主循环里持续轮询 IBUS
```

关键函数作用：

`uart3_read_byte()` 直接读 UART3 寄存器。它先看 `UART_LSR_DR`，如果 FIFO 有数据，再读 `UART_RBR`。  
`ibus_parse_byte()` 按 iBUS 协议收 32 字节帧，帧头是 `0x20 0x40`，最后两个字节是 checksum。  
`ibus_probe_poll_once()` 是现在最重要的轮询函数，它读 IBUS、定时读传感器、更新 RC、安全门控、控制、电机和 debug shm。

当前通道语义：

```text
ch[0] = roll 横滚输入
ch[1] = pitch 俯仰输入
ch[2] = throttle 总推力输入
ch[3] = yaw 偏航角速度输入
ch[5] = arm switch 解锁开关
```

这里 throttle 不是直接控制某一个电机转速，而是 collective thrust，也就是总推力意图。四个电机最终输出要经过 PID 和 mixer。

## 4. MPU6050 和 HMC5883L 读取

### 4.1 bit-bang I2C

代码位置：

```text
comm_main.c
bb_i2c_start()
bb_i2c_stop()
bb_i2c_write_bit()
bb_i2c_read_bit()
bb_i2c_write_byte()
bb_i2c_read_byte()
bb_i2c_read_reg()
bb_i2c_write_reg()
bb_i2c_read_regs()
```

当前没有用 Linux i2c0，也没有用 RTOS HAL I2C0。GP0/GP1 被当成 GPIO，用代码模拟 I2C。I2C 是开漏总线，所以代码里拉低时用 `bb_gpio_output_low()`，拉高时不是强推高电平，而是 `bb_gpio_release()` 释放引脚，让外部上拉电阻把线拉高。

### 4.2 MPU6050

代码位置：

```text
comm_main.c
mpu6050_config_once()
mpu6050_read_raw_once()
mpu6050_cal_update_once()
sensor_update_corrected_once()
```

`mpu6050_config_once()` 尝试 MPU 地址 `0x68/0x69`，检查 `WHO_AM_I=0x68`，然后配置：

```text
PWR_MGMT_1      唤醒 MPU
SMPLRT_DIV      采样分频
CONFIG          低通滤波
GYRO_CONFIG     陀螺仪量程
ACCEL_CONFIG    加速度计量程
```

`mpu6050_read_raw_once()` 从 `ACCEL_XOUT_H` 开始读 14 字节，得到：

```text
g_mpu_ax, g_mpu_ay, g_mpu_az
g_mpu_gx, g_mpu_gy, g_mpu_gz
```

`mpu6050_cal_update_once()` 做静止校准，计算 gyro bias 和 acc bias。  
`sensor_update_corrected_once()` 用 bias 修正 gyro：

```text
g_mpu_gx_corr = g_mpu_gx - g_mpu_gyro_bias_x
```

并更新：

```text
g_sensor_ready
g_att_ok
g_sensor_state_error
```

当前问题：传感器读取周期在 `ibus_probe_poll_once()` 里是 `pdMS_TO_TICKS(20)`，约 50Hz，还没达到老师要求的 100Hz。

### 4.3 HMC5883L

代码位置：

```text
comm_main.c
hmc5883l_config_once()
hmc5883l_read_raw_once()
```

HMC 地址是 `0x1E`，ID 应为：

```text
0x48, 0x34, 0x33
```

`hmc5883l_config_once()` 配置连续测量模式。  
`hmc5883l_read_raw_once()` 读取磁力计 X/Y/Z，保存到：

```text
g_hmc_x
g_hmc_y
g_hmc_z
g_hmc_data_ok
```

HMC 当前只用于 yaw / heading debug。当前没有做 yaw heading hold，所以不能说“磁力计已经参与偏航闭环”。

## 5. 姿态解算现在到底是什么

代码位置：

```text
comm_main.c
attitude_update_debug_once()   姿态 debug 解算
body_axis_update_once()        转换到机体系并做 level trim
amp_abs32_to_u32()             整数绝对值
amp_isqrt64()                  整数平方根
amp_atan2_cd_approx()          atan2 的整数近似
```

当前姿态解算不是完整互补滤波。它现在做的是：

```text
roll/pitch：由加速度计估算
yaw：由 HMC5883L 磁力计估算
rate_raw：直接来自 gyro_corr
```

roll/pitch 原理是：静止时加速度计主要测到重力方向，所以可以由 `ax/ay/az` 算倾角：

```text
roll  = atan2(ay, az)
pitch = atan2(-ax, sqrt(ay^2 + az^2))
```

代码没有用 `atan2f()` 和 `sqrtf()`，而是用整数近似函数，避免 RTOS 小核链接 libm。角度单位是 centi-degree：

```text
100 = 1 度
```

`body_axis_update_once()` 做 level trim。因为上电时机体水平放置，所以第一次有效姿态被记录为：

```text
g_level_roll_cd
g_level_pitch_cd
```

后续机体系角度是：

```text
g_body_roll_cd  = g_att_roll_cd  - g_level_roll_cd
g_body_pitch_cd = g_att_pitch_cd - g_level_pitch_cd
```

如果安装方向反了，只改：

```text
BODY_ROLL_SIGN
BODY_PITCH_SIGN
BODY_YAW_SIGN
```

不要直接改 mixer 公式。

## 6. 互补滤波和卡尔曼滤波是什么，为什么现在还没做完整

### 6.1 互补滤波

MPU6050 里有加速度计和陀螺仪。加速度计长期不漂，但动态时会抖；陀螺仪动态响应快，但积分会漂。互补滤波就是把两者合起来：

```text
gyro_angle = last_angle + gyro_rate * dt
fused_angle = alpha * gyro_angle + (1 - alpha) * accel_angle
```

`alpha` 通常接近 1，例如 0.98。意思是短时间主要相信 gyro，长时间用 accel 慢慢修正漂移。

yaw 方向不能靠加速度计修正，需要 HMC：

```text
yaw_gyro = last_yaw + gyro_yaw_rate * dt
yaw = alpha * yaw_gyro + (1 - alpha) * mag_yaw
```

当前没有完整实现互补滤波，是因为我们先把硬件链路、PWM、电机映射、安全门控、body_rate、mixer 都验证通。否则一上来写复杂滤波，出了问题不知道是传感器、PWM、mixer 还是算法错。

后续应该把互补滤波放到：

```text
当前 bring-up：comm_main.c 新增 estimator_update_once()
正式框架：rt_core/Estimator.hpp / rt_core/Estimator.cpp
```

### 6.2 卡尔曼滤波

卡尔曼滤波比互补滤波更复杂。它不是简单按比例混合，而是维护一个状态，例如：

```text
angle
gyro_bias
```

预测阶段用 gyro：

```text
angle = angle + (gyro_rate - gyro_bias) * dt
```

修正阶段用 accel 或 mag：

```text
error = measured_angle - predicted_angle
```

然后根据噪声协方差更新 angle 和 bias。优点是理论上更稳，还能估计 gyro bias；缺点是计算和调参复杂。

正式框架里其实有卡尔曼入口：

```text
rt_core/Estimator.hpp
AxisKalman
Estimator
```

所以不是 `amp_fc` 框架没有卡尔曼，而是当前 `comm_main.c` 还没接进去。当前阶段接太早容易把已经验证的硬件链路弄乱。

## 7. PID：位置式、增量式，以及我现在用的是哪种

### 7.1 位置式 PID

位置式 PID 直接计算当前输出值：

```text
u(k) = Kp * e(k) + Ki * sum(e) + Kd * (e(k) - e(k-1))
```

它输出的是“当前控制量应该是多少”。优点是直观，适合姿态角、角速度这种闭环；缺点是积分项如果没限幅，可能 windup。

### 7.2 增量式 PID

增量式 PID 计算的是输出变化量：

```text
delta_u = Kp * (e(k) - e(k-1))
        + Ki * e(k)
        + Kd * (e(k) - 2e(k-1) + e(k-2))

u(k) = u(k-1) + delta_u
```

它适合一些执行器控制或需要平滑变化的场景。优点是输出变化相对平滑；缺点是要维护历史输出和历史误差，调试时不如位置式直观。

### 7.3 当前代码中的 PID

代码位置：

```text
comm_main.c
control_update_setpoint_once()
control_update_pid_once()
control_update_mixer_once()
```

当前更接近“位置式 P-only rate controller”。它没有积分和微分，也没有计算 delta_u。核心是：

```text
roll_rate_err  = g_sp_roll_rate_raw  - g_body_rate_roll_raw
pitch_rate_err = g_sp_pitch_rate_raw - g_body_rate_pitch_raw
yaw_rate_err   = g_sp_yaw_rate_raw   - g_body_rate_yaw_raw
```

然后：

```text
g_rate_corr_roll  = roll_rate_err  / RATE_PID_ROLL_DIV
g_rate_corr_pitch = pitch_rate_err / RATE_PID_PITCH_DIV
g_rate_corr_yaw   = yaw_rate_err   / RATE_PID_YAW_DIV
```

再限幅到：

```text
RATE_PID_MAX_CORR_US
```

roll/pitch 还有 angle assist：

```text
g_pid_roll  = rate_corr_roll  + angle_corr_roll
g_pid_pitch = rate_corr_pitch + angle_corr_pitch
g_pid_yaw   = rate_corr_yaw
```

当前不用完整 PID 的原因是安全和调试顺序。我们现在还在确认 body axis、mixer 符号、电机映射和传感器方向。如果这时候加 I/D，输出变化会更难判断，甚至可能因为积分累积导致突然大输出。

## 8. 为什么先调内环，再调外环

无人机控制一般是级联结构：

```text
外环：姿态角控制
  目标角度 - 当前角度 -> 目标角速度

内环：角速度控制
  目标角速度 - 当前角速度 -> 电机修正量
```

内环必须先稳定，因为电机真正能快速影响的是角速度。gyro 的 body_rate 反馈快，适合先判断电机响应方向。如果内环都不稳，外环给出的角速度目标再合理也没用，最后表现就是机体乱抖或方向反。

当前我们进行到的位置是：

```text
内环 rate PID 小限幅已经接入
angle assist 只是辅助观察静态倾斜
还没有完整 angle 外环
还没有完整互补/卡尔曼 Estimator
还没有有桨悬停调参
```

所以现在应该先验证：

```text
摇 roll 杆 -> sp_roll_rate_raw 变化 -> pid_roll 变化 -> mix/motor 差速
快速摆 MPU -> body_rate 变化 -> rate_corr 变化 -> mix/motor 差速
轻微静态倾斜 -> body_roll/body_pitch 变化 -> angle_corr 变化
```

这些都对了，才继续做完整外环 angle PID。

## 9. 安全门控

安全相关函数：

```text
motor_set_all()
motor_force_safe()
rc_update_freshness_once()
motor_update_arm_gate()
ibus_probe_poll_once() 中的 safety gate
```

`motor_force_safe()` 会撤销 arm，清 output enabled，记录原因，并强制：

```text
motor=1000,1000,1000,1000
```

safe reason 包括：

```text
MOTOR_SAFE_FAILSAFE
MOTOR_SAFE_SENSOR_NOT_RDY
MOTOR_SAFE_NOT_ARMED
MOTOR_SAFE_THROTTLE_HIGH
```

`rc_update_freshness_once()` 超过 `RC_LOST_TIMEOUT_MS=300ms` 没有新帧，就认为 RC lost。  
`motor_update_arm_gate()` 要求上电后先看到 arm switch low，再 low->high，防止接收机保持旧的 arm 高电平导致自动解锁。  
`RC_THROTTLE_TEST_KILL_US=1350` 用于油门过高时撤销 arm。

safe 分支不能加 trim/min_spin。因为 trim/min_spin 会把电机提高到起转附近，而 safe 的语义就是必须停桨。

## 10. 我们实际踩过的坑

这些是报告里可以写成“调试过程”的内容，比空泛讲架构更真实。

1. 一开始 HMC/MPU 不稳定，所以没有继续走 Linux i2c0 或 RTOS HAL I2C0，而是改成 GP0/GP1 C906 bit-bang I2C，最终读通 `hmc_id=48,34,33` 和 `mpu_whoami=68`。

2. PWM 最开始只显示 ready，但 ESC 只滴滴响不转。后来确认 `pwm_configuration_t.period/pulse` 单位是 ns，不是 tick，所以修正为 `us * 1000`。

3. SDK HAL 里 `pwm_set_output_cfg()` 可能影响 PWMSTART，因此现在每次写 PWM 后都重新调用 `pwm_continuous_start()`，并读回 `pwm_start/pwm_oe`。

4. GP2/PWM7 和 GP3/PWM6 能转，GP10/PWM10、GP11/PWM11 读回看似正常但 ESC 不识别，后来放弃这两路。

5. GP6/PWM9 测试也不理想，最后使用同一个 PWM bank 的 PWM7/PWM5/PWM4/PWM6。

6. 曾经 `amp_dbg_dump.c` 结构体和 `comm_main.c` 的 `amp_dbg_shm_t` 不一致，导致 `hmc_addr`、`rc_valid`、`thr` 打印乱码。后来同步字段顺序解决。

7. 遥控器关闭或接收机保持旧通道值时，曾出现 arm 状态不安全，所以加入 `g_rc_seen_arm_low`、`g_rc_arm_latched`，要求先看到 arm low，再 low->high 才能解锁。

8. 当前 HMC yaw 只作为 debug，不做 heading hold。这个不能夸大，否则老师问 yaw 闭环会露馅。
